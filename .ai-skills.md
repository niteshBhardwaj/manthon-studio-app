# Manthan Studio - AI Skills Reference

## Project Overview

Manthan Studio is an Electron + React + TypeScript desktop app for AI media generation.
The app now uses a model-driven generation system:

- models are defined in a shared registry
- prompt controls are rendered from model capabilities
- models can be enabled or disabled from settings
- API keys can be shared across related providers through key groups
- generation requests now flow through a persistent, prioritized job queue
- storage usage and retention policies are managed centrally in the main process

## Tech Stack

- Framework: `electron-vite` (Electron + Vite + React 19 + TypeScript)
- Styling: Tailwind CSS v4
- State: Zustand
- API: `@google/genai`
- Icons: `lucide-react`
- Animations: `framer-motion`

## Architecture Patterns

### 1. Process Separation

```text
src/main/      -> Node main process (providers, storage, IPC)
src/preload/   -> secure contextBridge API
src/renderer/  -> React UI
src/shared/    -> cross-process shared registry/types
```

Rule: never call provider SDKs from the renderer.

### 2. Shared Model Registry

The source of truth for models lives in:

- `src/shared/model-registry.ts`

Each model declares:

- `id`
- `provider`
- `keyGroup`
- `contentType`
- `capabilities`
- optional `modes`
- optional `supportsVideoExtension` (video models only — gates true Veo video-to-video extension in the lightbox)

Renderer code consumes this via:

- `src/renderer/src/lib/model-capabilities.ts`

Main process code also consumes the shared registry for:

- provider-group mapping
- grouped key initialization
- enabled-model defaults

Storage, retention, and disk inspection stay in `src/main/store/` and should not be implemented in the renderer.

### 3. Dynamic Prompt Architecture

The prompt surface is no longer hardcoded around specific providers.

Current flow:

1. selected content type chooses enabled models for that type
2. selected model comes from the shared registry
3. capabilities are rendered dynamically from the model descriptor
4. generic capability values are converted into provider payloads by `build-payload.ts`

Important renderer files:

- `src/renderer/src/components/generation/PromptInput.tsx`
- `src/renderer/src/lib/build-payload.ts`
- `src/renderer/src/stores/generation-store.ts`
- `src/renderer/src/stores/model-store.ts`

### 4. Grouped API Key System

Keys are no longer only managed per provider.

Current behavior:

- models declare a `keyGroup` such as `google`
- settings shows one key input per group
- saving a group key initializes every provider in that group
- legacy provider-level keys are migrated lazily into group storage

Main files:

- `src/main/store/key-store.ts`
- `src/main/providers/registry.ts`
- `src/main/ipc/index.ts`

Renderer entry:

- `src/renderer/src/components/settings/ApiKeyManager.tsx`

### 5. State Management

Primary stores:

- `useAppStore` - shell UI state, modals, search, toast queue, history pulse, media UX preferences
- `useGenerationStore` - prompt text, selected model, content type, capability values, extend-video state, completed media shown on the Create canvas
- `useModelStore` - enabled model IDs and persistence
- `useProviderStore` - provider connection state and active provider status
- `useQueueStore` - persistent queue state, pause/resume, retry, cancel, reorder

`useGenerationStore` now uses:

- `contentType`
- `selectedModel`
- `activeMode`
- `batchCount`
- `capabilityValues`
- `extendingJobId`
- `videoInput` — holds the full video binary for true Veo extension (null otherwise)

instead of hardcoded fields like `aspectRatio`, `resolution`, and `enableAudio`.

### 6. IPC Pattern

Typical flow:

```typescript
// preload
contextBridge.exposeInMainWorld('manthan', {
  getEnabledModels: () => ipcRenderer.invoke('models:get-enabled'),
  setEnabledModels: (ids) => ipcRenderer.invoke('models:set-enabled', ids),
  saveGroupKey: (group, key) => ipcRenderer.invoke('api-key:save-group', group, key)
})

// main
ipcMain.handle('models:get-enabled', async () => appStore.getEnabledModels())

// renderer
const ids = await window.manthan.getEnabledModels()
```

## Common Tasks

### Add a new model

1. Add the model to `src/shared/model-registry.ts`
2. Point it at an existing or new provider ID
3. Define its `contentType`, `capabilities`, and optional `modes`
4. If needed, update `build-payload.ts` so generic capability values map to provider params
5. If it belongs to a new key family, add a new key group

In most cases you do not need to hardcode any new UI in `PromptInput.tsx`.

### Add a new provider

1. Implement `MediaProvider` in `src/main/providers/`
2. Register it in `src/main/providers/registry.ts`
3. If models for it should share a key, connect it to a `keyGroup` in `src/shared/model-registry.ts`
4. Expose any new IPC methods in preload types and bridge

### Add a new dynamic capability

1. Extend `CapabilityType` in `src/shared/model-registry.ts`
2. Add descriptor data to the relevant models
3. Teach `PromptInput.tsx` how to render that capability (most capabilities auto-render from `CapabilityRenderer`)
4. Teach `build-payload.ts` how to map it into provider params
5. Wire the provider param field in `base.ts` (`VideoGenParams` / `ImageGenParams` / `AudioGenParams`)
6. Send the param to the API in the provider implementation

Example: `duration_seconds` → mapped in build-payload → `VideoGenParams.durationSeconds` → sent as `genConfig.durationSeconds` by google-veo provider

### Add a new IPC channel

1. Add handler in `src/main/ipc/index.ts`
2. Expose it in `src/preload/index.ts`
3. Add types in `src/preload/index.d.ts`
4. Call it through `window.manthan`

### Add or change queued generation behavior

1. Update enqueue payload building in `src/renderer/src/lib/enqueue-generation.ts`
2. Keep queue job payloads serializable and restart-safe
3. Update dispatch or polling logic in `src/main/queue/job-processor.ts`
4. Update persistence, retries, and notifications in `src/main/queue/queue-manager.ts`
5. Reflect any new queue events in `src/renderer/src/stores/queue-store.ts`

### Add or change storage policy behavior

1. Update retention or disk logic in `src/main/store/storage-manager.ts`
2. Keep policy values serialized through the `preferences` table, not renderer-only state
3. Expose new storage methods through `src/preload/index.ts` and `src/preload/index.d.ts`
4. Surface policy changes in `src/renderer/src/components/settings/StorageDashboard.tsx`

## Known Gotchas

- `window.manthan` must still be guarded in browser-only dev situations
- video generation remains async and relies on polling
- grouped key tests initialize a provider instance temporarily, similar to the old per-provider key test flow
- the top bar now reflects the currently selected model; it should not become a second source of truth for model selection
- batch count currently fans out into multiple generation jobs rather than one grouped multi-result job; Veo always uses `numberOfVideos: 1`
- `safeStorage` may be unavailable in some dev environments, so key storage falls back to base64 encoding
- queue processing is authoritative in the main process; renderer components should not try to run providers directly
- Veo 3.1 audio is always on — there is no `enableAudio` toggle; `audio_toggle` was removed from the model registry
- Veo video extension requires the video to have been generated by Veo within the last 2 days; the API does not support extending uploaded/imported videos
- Veo `duration_seconds` only supports values `4`, `6`, `8` (not arbitrary); `8s` is required for 1080p, 4K, and reference images

### Video Extension Architecture

The lightbox offers two distinct video continuation workflows with smart visibility:

#### Extend Video (+7s) — True Veo Extension

- Sends the full video binary via `params.video` to the Veo extension API
- Resolution is locked to `720p`, `numberOfVideos` always `1`
- Only available for Veo-generated videos from models with `supportsVideoExtension: true` (currently `veo-3.1-generate-preview` and `veo-3.1-fast-generate-preview`)
- Input video must be ≤141 seconds; can extend up to 20 times (max 148s)
- Output is a single combined video (original + new 7s)

#### Continue from Last Frame — Image-to-Video

- Extracts the last frame via `video-utils.ts` and creates a new image-to-video generation
- Available for any video source (Veo-generated, uploaded, imported)
- All standard capabilities apply (aspect ratio, resolution, duration, etc.)

#### Visibility Rules

| Video Source | Extend Video | Continue from Last Frame |
|---|---|---|
| Veo 3.1 generated | ✅ | ✅ |
| Veo 3.1 Fast generated | ✅ | ✅ |
| Veo 3.1 Lite generated | ❌ | ✅ |
| Uploaded / Imported | ❌ | ✅ |

#### Key state:

- `useGenerationStore.extendingJobId` signals extend mode in `PromptInput.tsx`
- `useGenerationStore.videoInput` holds the full video binary for true Veo extension
- extend-video should inherit the original job's model and capability values
- new jobs created from either mode go through the durable queue path

#### Important files:

- `src/renderer/src/components/player/VideoLightbox.tsx` — dual mode UI with animated panels
- `src/renderer/src/lib/video-utils.ts` — `extractFrameAtTime()` for last frame capture
- `src/renderer/src/lib/enqueue-generation.ts` — accepts `videoInput` and passes as `video-extend` input asset

## Database & Storage Layer (Phase 1)

### SQLite Integration

The app uses `better-sqlite3` for persistent storage, replacing the old flat JSON file stores.

Database location: `{userData}/manthan.db` (e.g., `%APPDATA%/manthan-studio/manthan.db` on Windows)

Key files:

- `src/main/store/db.ts` — `DatabaseManager` singleton (init, query helpers, migrations, shutdown)
- `src/main/store/schema.ts` — versioned SQL migration definitions
- `src/main/store/app-store.ts` — app preferences & generation history (backed by SQLite)
- `src/main/store/asset-manager.ts` — managed media file storage + DB catalog

### Database Architecture

```text
DatabaseManager (singleton)
├── initialize()     → creates/opens manthan.db, runs migrations
├── getDb()          → raw better-sqlite3 Database instance
├── query<T>()       → SELECT returning rows
├── queryOne<T>()    → SELECT returning single row
├── run()            → INSERT/UPDATE/DELETE
├── transaction()    → wrap multiple statements atomically
└── close()          → WAL checkpoint + close connection
```

Pragmas set on initialization:
- `journal_mode = WAL` (write-ahead logging for concurrent reads)
- `foreign_keys = ON`
- `busy_timeout = 5000`

### Schema Tables

| Table | Purpose | Phase |
|-------|---------|-------|
| `projects` | Workspace containers | 2 |
| `generations` | Generation history (replaces JSON history) | 1 |
| `assets` | Unified media file catalog | 1 |
| `preferences` | Key-value settings store | 1 |
| `prompt_templates` | Reusable prompt templates | 1 (seeded), 3 (enhanced) |
| `variables` | Prompt variables (`{var_name}` system) | 3 |
| `cost_log` | API spend tracking per generation | 6 |
| `job_queue` | Persistent generation job queue | 4 |
| `negative_prompt_presets` | Reusable negative prompt presets | 3 |
| `_migrations` | Migration version tracking (internal) | 1 |

All tables are created in migration v1 (forward-looking). Future phases use existing tables.

### Migration System

Migrations are versioned and tracked in the `_migrations` table:

```typescript
// schema.ts
export const MIGRATIONS: Migration[] = [
  { version: 1, description: 'Initial schema', sql: '...' },
  // Future: { version: 2, description: 'Add columns for X', sql: '...' }
]
```

On `initialize()`, the manager checks which versions have been applied and runs any missing ones inside transactions.

### JSON → SQLite Migration

On first run after the Phase 1 update:

1. Detects `manthan-app.json` in userData
2. If `generations` table is empty, imports all history rows + preferences + templates
3. Renames old file to `manthan-app.json.bak`

This is handled by `appStore.initialize()` which must be called after `databaseManager.initialize()`.

### Asset Manager

Manages media files on disk with a catalog in SQLite.

Directory structure:
```text
{userData}/assets/{project_id}/{asset_id}.{ext}
{userData}/cache/   (temp files, clearable)
```

Key methods:

| Method | Purpose |
|--------|---------|
| `saveAsset(opts)` | Write buffer to disk + create DB row |
| `saveBase64Asset(opts)` | Decode base64 + delegate to saveAsset |
| `importFromFile(projectId, path)` | Copy external file into managed storage |
| `getAsset(id)` | Get asset metadata from DB |
| `listAssets(opts)` | Filtered, paginated asset listing |
| `readAssetBase64(id)` | Read file data for IPC transfer |
| `deleteAsset(id)` | Remove file + DB row |
| `getStorageStats()` | Disk usage breakdown by type |
| `cleanupCache()` | Clear cache dir, return freed bytes |

### Asset IPC Channels

| Channel | Preload Method | Description |
|---------|---------------|-------------|
| `asset:save` | `saveAsset(opts)` | Save base64 data as managed asset |
| `asset:list` | `listAssets(opts?)` | List assets with filters |
| `asset:get` | `getAsset(id)` | Get single asset metadata |
| `asset:read` | `readAsset(id)` | Read asset file as base64 |
| `asset:delete` | `deleteAsset(id)` | Delete asset (file + DB) |
| `asset:import` | `importAssets(projectId?)` | Open file dialog + import |
| `asset:stats` | `getStorageStats()` | Storage breakdown by type |
| `asset:cleanup-cache` | `cleanupCache()` | Clear cache, return freed bytes |
| `storage:open-folder` | `openStorageFolder()` | Open userData in file explorer |

### Initialization Order

In `src/main/index.ts`, the initialization must follow this order:

```typescript
app.whenReady().then(async () => {
  databaseManager.initialize()    // 1. Open/create DB, run migrations
  assetManager.initialize()       // 2. Create asset directories
  storageManager.initialize()     // 3. Create cache/temp directories for storage tools
  appStore.initialize()           // 4. Migrate JSON data, seed defaults
  registerIpcHandlers()           // 5. Register all IPC channels
  await initializeStoredProviders() // 6. Load API keys, init providers
  queueManager.initialize()       // 7. Resume persistent queue processing
  createWindow()                  // 8. Open the window
})
```

Shutdown:
```typescript
app.on('before-quit', () => {
  databaseManager.close()  // WAL checkpoint + close
})
```

### Storage Separation

| Data | Storage | Reason |
|------|---------|--------|
| API keys | `JsonStore` + `safeStorage` | OS-level encryption required |
| History, preferences, templates | SQLite (`manthan.db`) | Relational, queryable |
| Media files | Filesystem (`assets/`) | Large binary blobs |
| Media catalog | SQLite (`assets` table) | Metadata, filtering, stats |

**Important**: API keys intentionally stay in `key-store.ts` using `JsonStore` + Electron `safeStorage`. They must NOT be moved to SQLite.

## Resource & Disk Space Manager (Phase 5)

### Architecture

Storage visibility and retention policy execution now live in:

- `src/main/store/storage-manager.ts` - filesystem scanning, disk info lookup, cache cleanup, retention execution
- `src/renderer/src/components/settings/StorageDashboard.tsx` - storage chart, disk bar, cleanup actions, policy controls

The settings modal now has three major sections:

- Models
- Keys
- Storage

### Storage Responsibilities

`StorageManager` is responsible for:

- walking `userData/assets/` and grouping file usage by media type
- counting `cache/` and `temp/` storage
- including SQLite database size in totals
- reading and writing retention policy preferences
- deleting old un-starred generations and their associated assets
- checking system disk space from the main process

### Retention Policy Shape

```typescript
interface RetentionPolicy {
  deleteUnstarredAfterDays: number | null
  clearCacheOnExit: boolean
  maxStorageMB: number | null
}
```

Policy values are stored under a single preferences entry and should always remain serializable.

### Storage IPC Channels

| Channel | Purpose |
|--------|---------|
| `storage:breakdown` | full storage report including per-type table data |
| `storage:disk-info` | system disk totals and free space |
| `storage:cleanup-cache` | clear `cache/` and `temp/` files |
| `storage:get-policy` | load retention policy |
| `storage:set-policy` | persist retention policy |
| `storage:apply-policy` | execute cleanup immediately |
| `storage:open-folder` | open app storage directory in the OS file explorer |

### Storage Safety Notes

- Use the main process for filesystem walking and deletion; the renderer should only display results and trigger actions.
- Retention cleanup should respect `starred = 1` rows even if they are old.
- New completed generations now persist `result_asset_id`, which makes asset cleanup more reliable than older history rows.

## Known Gotchas (Updated)

- `window.manthan` must still be guarded in browser-only dev situations
- video generation remains async and relies on polling
- grouped key tests initialize a provider instance temporarily, similar to the old per-provider key test flow
- the top bar now reflects the currently selected model; it should not become a second source of truth for model selection

- **Asset previews**: The asset grid uses `asset://` protocol URLs for image previews. This requires a custom protocol handler to be registered in the main process (not yet implemented) — images will show placeholder icons until then.

## Project Workspaces (Phase 2)

### Architecture

Projects are the top-level isolation unit. Every asset, generation, variable, and template is scoped to a project via `project_id` foreign keys.

```text
ProjectManager (main process singleton)
├── createProject(opts)     → INSERT + return project
├── listProjects()          → SELECT with JOINed generation_count + asset_count
├── updateProject(id, ups)  → partial UPDATE (name, description, color, icon)
├── archiveProject(id)      → soft delete (archived = 1)
├── deleteProject(id)       → hard delete + cascade (assets, generations, variables, templates)
└── getColors()             → 8 preset color palette
```

Key files:

- `src/main/store/project-manager.ts` — CRUD operations on `projects` table
- `src/renderer/src/stores/project-store.ts` — Zustand store with `activeProjectId`
- `src/renderer/src/components/layout/ProjectSwitcher.tsx` — TopBar dropdown UI

### Project Store (Renderer)

```typescript
interface ProjectState {
  projects: Project[]
  activeProjectId: string      // defaults to 'default'
  loading: boolean
  loadProjects: () => Promise<void>
  createProject: (name, color?) => Promise<Project | null>
  updateProject: (id, updates) => Promise<void>
  deleteProject: (id) => Promise<void>

```

Key behaviors:
- `createProject` auto-switches to the newly created project
- `deleteProject` falls back to `'default'` if the active project was deleted

### How Project Scoping Works

When adding new features that are project-scoped:

1. Add `project_id TEXT REFERENCES projects(id)` to the table
2. Filter queries by `activeProjectId` in the renderer store or IPC handler
3. Use `useProjectStore().activeProjectId` in React components
4. Handle the `'default'` project as fallback

Example pattern:
```typescript
// Renderer: pass activeProjectId to IPC
const { activeProjectId } = useProjectStore()
const result = await window.manthan.listAssets({ projectId: activeProjectId })

// Main: filter by project_id in SQL
const rows = db.query('SELECT * FROM assets WHERE project_id = ?', [projectId])
```

## Advanced Queue Manager (Phase 4)

### Architecture

Generation no longer executes inline in IPC. All `gen:video`, `gen:image`, and `gen:audio` requests now enqueue persistent jobs in `job_queue`.

Main-process queue files:

- `src/main/queue/queue-manager.ts` - singleton orchestrator, DB persistence, retries, pause/resume, notifications, IPC event fanout
- `src/main/queue/job-processor.ts` - provider dispatch, video polling loop, timeout handling, progress callbacks
- `src/main/queue/types.ts` - shared queue job/event/state types
- `src/main/notifications.ts` - native desktop notifications gated by `preferences.notificationsEnabled`

Renderer queue files:

- `src/renderer/src/stores/queue-store.ts` - queue state subscription and commands
- `src/renderer/src/components/queue/QueueDashboard.tsx` - Running / Pending / Completed / Failed dashboard

### Queue Flow

1. `PromptInput.tsx` builds provider params and enqueues one durable job per requested output
2. Main IPC inserts into `job_queue` through `queueManager.enqueue()`
3. `QueueManager` processes pending jobs with per-type concurrency limits
4. `JobProcessor` calls the provider and emits progress
5. Completion persists the generated asset, updates history, and emits queue events
6. Renderer updates `useQueueStore` in real time and mirrors completed jobs into `useGenerationStore`

### Queue IPC Channels

| Channel | Purpose |
|--------|---------|
| `queue:list` | full queue snapshot |
| `queue:pause` | pause processing |
| `queue:resume` | resume processing |
| `queue:cancel` | cancel running or pending job |
| `queue:retry` | requeue failed/cancelled job |
| `queue:reorder` | update pending-job priority |
| `queue:clear-completed` | remove completed jobs from queue history |
| `queue:delete` | remove a specific non-running job |

### Queue Events

| Event | Payload |
|------|---------|
| `queue:update` | full `QueueState` snapshot |
| `queue:job-progress` | `{ jobId, progress, status }` |
| `queue:job-complete` | `{ jobId, job, result }` |
| `queue:job-failed` | `{ jobId, job, error }` |

### Queue Ownership Rules

- The main process is the source of truth for queue state.
- `useQueueStore` owns pending/running/failed/completed queue metadata.
- `useGenerationStore` should only be used for the Create-canvas media gallery and prompt state.
- If you add a new generation capability, make sure its provider params are preserved inside the queue job config so restart recovery keeps working.

## Asset Picker Modal

### Architecture

Replaces the raw system file-open dialog with a rich, project-aware Asset Picker modal for prompt media attachments.
Allows browsing, filtering, searching, and previewing assets from any workspace project before attaching them.

Key files:
- `src/renderer/src/hooks/useAssetPicker.ts` - State management for browsing assets, client-side filtering and sorting.
- `src/renderer/src/components/generation/AssetPickerModal.tsx` - Modal interface featuring project dropdown, search bar, sort options, media type chips, thumbnail grid, large preview, and multi-select.
- `src/renderer/src/components/generation/PromptInput.tsx` - Modified to open the picker and convert selected assets to `BinaryInput` for attachment.

### Flow
1. User clicks the attachment button in `PromptInput`.
2. `AssetPickerModal` opens.
3. User selects one or multiple assets, optionally filtering by project or search query.
4. Preview shows large image/video poster/audio player and asset metadata.
5. "Upload new" allows direct upload via system file dialog, automatically saving to the selected project.
6. "Attach" reads the selected assets' file contents via `window.manthan.readAsset(id)` and sets them as start frame, end frame, or reference images in the generation store.
